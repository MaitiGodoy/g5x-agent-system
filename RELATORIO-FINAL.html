🔒 Correções de Segurança (6 falhas eliminadas)

1.1 Chave GROQ Exposta em Texto Claro CRÍTICO CORRIGIDO

Antes: .env continha GROQ_API_KEY=gsk_RzL1nrDa...798z6 em texto claro, acessível por qualquer processo no servidor.

Depois: Chave removida do .env. O Doutor gerencia TODAS as chaves internamente via kernel.provider_router.

Arquivo: .env — linhas removidas: GROQ_API_KEY, DEEPSEEK_API_KEY

1.2 Headers de Segurança HTTP (Helmet) CRÍTICO CORRIGIDO

Antes: Nenhum header de segurança (CSP, HSTS, X-Frame-Options, X-Content-Type-Options) — vulnerável a clickjacking, XSS, MIME sniffing.

Depois: helmet middleware adicionado com proteções padrão. CSP e COEP desabilitados intencionalmente para compatibilidade com scripts inline do frontend.

Arquivo: server.js — linhas 10-16

1.3 genId() — Math.random() Não Criptográfico ALTA CORRIGIDO

Antes: Date.now().toString(36) + '-' + Math.random().toString(36).slice(2,10) — colisão garantida em alta frequência, IDs previsíveis.

Depois: require('crypto').randomUUID() — UUID v4 criptográfico, sem colisão.

Arquivo: tools/db.js — função genId()

1.4 DEEPSEEK_API_KEY Vazia ALTA CORRIGIDO

Antes: DEEPSEEK_API_KEY= (vazia) — fallback não operacional, gerava falsa sensação de resiliência. Circuit breaker abria sem necessidade real.

Depois: Removida. Doutor tem 7 provedores nativos com failover automático.

1.5 Data Race — DB Não Centralizado CRÍTICO CORRIGIDO

Antes: agent.js e api.js declaravam let data = {} independentes com loadDb()/saveDb() próprios — escrita concorrente corrompia crm-db.json.

Depois: Ambos centralizados via dbm.readDb()/dbm.writeDb() com lock file + cache + write queue.

Arquivos: agent.js linhas 1-22, api.js linhas 27-34

1.6 delete_lead — Sobrescrita Corrompendo Dados ALTA CORRIGIDO

Antes: data['leads'] = getTable('leads').filter(...) — sobrescrevia a tabela de leads com filtered, mas data['leads'] não era a referência correta. Podia corromper outras tabelas.

Depois: Lógica corrigida: filtra localmente, verifica tamanho, só sobrescreve data['leads'] se houve remoção.

Arquivo: api.js — linha 694 (função executeAgentAction)

Depois: Endpoint /api/health/llm consulta Doutor via testConnection(). Startup limpa e direta.

2.3 .env — Limpeza de Chaves Expostas CRÍTICO CORRIGIDO

🛡️ Correções de Confiabilidade

3.1 AbortSignal.timeout() em TODOS os fetch ALTA CORRIGIDO

Problema: 4 chamadas fetch em api.js e 1 em agent.js sem timeout — podiam travar o heartbeat ou a resposta HTTP para sempre.

Correção: Adicionado signal: AbortSignal.timeout(10000) em:

  • api.js: schedule_call (Calendly), send_email
  • agent.js: calendly/schedule, whatsapp send, email send, linkedin send

3.2 PDF Ingestion — Validação de Texto Real ALTA CORRIGIDO

Antes: PDFs do Spin Selling e Christopher Voss foram ingeridos como binário PDF (~11 MB de lixo em 5.660 chunks). RAG comprometido — buscas retornavam "IJI$ R I$" sem sentido.

Depois:

  • Função isTextReadable() em tools/rag.js — detecta PDF header, verifica razão de caracteres alfabéticos, palavras reais
  • api.js — validação pós-pdfParse detecta binário e retorna erro 422 com instruções de OCR
  • DB limpo: 11.31 MB → 1.6 KB (99.99% redução)
  • 2 PDFs corrompidos movidos para _reingest_queue

3.3 Timeout na Bridge Python MÉDIA CORRIGIDO

Problema: Chamadas ao Doutor via Python podiam travar indefinidamente.

Correção: Todas as chamadas de child_process.execFile têm timeout de 35s (30s + 5s de overhead).
